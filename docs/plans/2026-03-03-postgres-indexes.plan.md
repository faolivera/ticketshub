# PostgreSQL indexes – Assessment and implementation plan

**Date**: 2026-03-03  
**Status**: Phase 1 done (high-priority indexes); Phase 2 pending (medium/low priority)

---

## 1. Context

The Prisma schema had no secondary indexes (`@@index`): only primary keys and unique constraints. Listings, transactions, notifications, and workers rely on filters and ordering that were causing full table scans as data grows.

This plan documents the read-pattern assessment, the indexes added in Phase 1, and the indexes left for future iterations.

---

## 2. Indexes added (Phase 1 – high priority)

The following indexes were added to the schema. Migration: `prisma/migrations/*_add_high_priority_indexes`. Index names in the database are auto-generated by Prisma unless set explicitly with `@@index([...], name: "…")`; the names in the tables below are for documentation only.

### 2.1 `ticket_listings`

| Index | Columns | Rationale |
|-------|---------|-----------|
| `ticket_listings_status_idx` | `(status)` | List active/pending/sold listings; catalog and admin flows |
| `ticket_listings_seller_id_idx` | `(sellerId)` | "My listings" by seller |
| `ticket_listings_event_id_idx` | `(eventId)` | Listings by event |
| `ticket_listings_event_date_id_idx` | `(eventDateId)` | Listings by event date |
| `ticket_listings_event_section_id_idx` | `(eventSectionId)` | Listings by section |
| `ticket_listings_event_id_status_idx` | `(eventId, status)` | Pending/active listings per event |
| `ticket_listings_event_date_id_status_idx` | `(eventDateId, status)` | Same per event date |
| `ticket_listings_event_section_id_status_idx` | `(eventSectionId, status)` | Same per section |
| `ticket_listings_status_created_at_idx` | `(status, createdAt(sort: Desc))` | Catalog ordered by newest |

### 2.2 `ticket_units`

| Index | Columns | Rationale |
|-------|---------|-----------|
| `ticket_units_listing_id_idx` | `(listingId)` | Units per listing; includes and reservations |
| `ticket_units_listing_id_status_idx` | `(listingId, status)` | Available units per listing (reservation flow) |

### 2.3 `transactions`

| Index | Columns | Rationale |
|-------|---------|-----------|
| `transactions_buyer_id_idx` | `(buyerId)` | "My purchases" |
| `transactions_seller_id_idx` | `(sellerId)` | "My sales" |
| `transactions_listing_id_idx` | `(listingId)` | Transactions per listing |
| `transactions_status_idx` | `(status)` | Counts and lists by status |
| `transactions_status_payment_expires_at_idx` | `(status, paymentExpiresAt)` | Cron: expired pending payments |
| `transactions_status_admin_review_expires_at_idx` | `(status, adminReviewExpiresAt)` | Cron: expired admin reviews |
| `transactions_created_at_idx` | `(createdAt(sort: Desc))` | Paginated lists by date |

### 2.4 `notifications`

| Index | Columns | Rationale |
|-------|---------|-----------|
| `notifications_recipient_id_channel_idx` | `(recipientId, channel)` | In-app inbox and count |
| `notifications_recipient_id_channel_read_idx` | `(recipientId, channel, read)` | Unread count; mark-all-read |
| `notifications_channel_status_idx` | `(channel, status)` | Email worker: PENDING / retry |
| `notifications_event_id_idx` | `(eventId)` | Notifications by event |

### 2.5 `notification_events`

| Index | Columns | Rationale |
|-------|---------|-----------|
| `notification_events_status_idx` | `(status)` | Worker: fetch PENDING events |
| `notification_events_status_triggered_at_idx` | `(status, triggeredAt(sort: Asc))` | Process PENDING in order |

## 3. Indexes pending (future iterations)

### 3.1 Medium priority

- **offers**: `(listingId)`, `(userId)`, `(listingId, status)`
- **events**: `(status)`, `(createdById)`, `(createdAt(sort: Desc))`
- **event_dates**: `(eventId)`, `(eventId, date)`, `(status)`
- **users**: `(role)` (if admin listing is hot)
- **wallet_transactions**: `(walletUserId)`, `(walletUserId, createdAt(sort: Desc))`
- **payment_confirmations**: `(status)` (pending queue)
- **pricing_snapshots**: `(listingId)`, `(expiresAt)` or composite for cleanup/consume
- **promotions**: `(userId)`, `(userId, type, status)`
- **otps**: `(userId, type, status)`
- **support_tickets**: `(userId)`, `(userId, status)`, `(transactionId)`
- **support_messages**: `(ticketId)`
- **identity_verification_requests**: `(userId)`, `(status)`
- **reviews**: `(transactionId)`, `(reviewerId)`, `(revieweeId)`

### 3.2 Low priority

- **scheduler_locks**: `(expiresAt)` (only if lock table grows)
- **notification_templates**: already covered by `@@unique([eventType, channel, locale])`

### 3.3 Optional / on demand

- **notifications**: `(channel, status, retryCount, nextRetryAt)` for retry worker; `(createdAt)` for `deleteOldNotifications` if table is very large.
- **transactions**: composite `(status, createdAt(sort: Desc))` if paginated lists by status become hot.

---

## 4. How to add pending indexes

1. Open `backend/prisma/schema.prisma`.
2. Add `@@index([...])` (and optionally named: `@@index([...], name: "optional_name")`) to the relevant model.
3. Run `npx prisma migrate dev --name add_<table>_indexes` (or a single migration for a batch).
4. Update this plan: move the added indexes from "Pending" to a new "Added in Phase 2" section with date.

---

## 5. Verification

After applying Phase 1 migration:

- Run `npx prisma migrate deploy` in target env.
- Optionally run `EXPLAIN (ANALYZE) ...` on hot queries (listings by event, transactions by buyer/seller, notification worker queries) to confirm index usage.
